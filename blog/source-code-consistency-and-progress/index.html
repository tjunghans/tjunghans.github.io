Code Consistency and Progress

Today I was in a discussion about using React Hooks in a large codebase, a 10 year old application that has evolved with time, or to stick with Classes and the custom service and state pattern.

There's a conflict between consistency and progress. Consistency is about having a coding style and patterns that allow the entire codebase to look and feel the same. This is great for maintenance and it looks neat. On the other hand, 100% consistency makes it impossible to introduce new ideas, such as patterns, libraries or even programming paradigms. Imagine having a JavaScript code base and not wanting to give Typescript a try, because it would make the codebase inconsistent, because updating 1000 js source files in one day is impossible. Imagine sticking to an imperative programming style and avoiding a declarative or functional style in favour of consistency. Consistency is great, but it does not allow for progress.
